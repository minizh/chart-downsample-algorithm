## 二、降采样算法性能

### 1. 折线图 (Line Chart)

#### 折线算法性能对比

| 指标 | LTTB 标准版 | MinMax 标准版 |
|------|------------|--------------|
| **原始数据** | 1,000,000 点 | 1,000,000 点 |
| **采样后** | 5,000 点 | 5,000 点 |
| **压缩比** | 200:1 | 200:1 |
| **采样耗时** | 876.10 ms | 820.70 ms |
| **渲染耗时** | 108.40 ms | 100.60 ms |
| **数据内存** | 0.2 MB | 0.2 MB |
| **原始数据内存** | 30.5 MB | 30.5 MB |

**内存变化趋势：**
- **LTTB**: `93MB → 235MB (+142MB) → 渲染完成后稳定在 110MB (60fps)`
- **MinMax**: `111MB → 216MB (+105MB) → 渲染完成后稳定在 110MB (60fps)`

**算法特性对比：**

| 特性 | LTTB 标准版 | MinMax 标准版 |
|------|------------|--------------|
| **核心策略** | 最大三角形三桶算法 | 分桶取极值（每桶保留 Min/Max） |
| **算法复杂度** | O(N) | O(N) |
| **首尾灯保留** | 强制保留 | 可选保留 |
| **适用场景** | 保持整体趋势和形状 | 关注极值范围和波动边界 |
| **性能特点** | 计算略慢，视觉保真度高 | 计算更快，极值保留完整 |

## 三、性能对比总结

| 图表类型 | 采样算法 | 压缩比 | 总耗时 | 内存峰值 | 渲染后内存 |
|---------|---------|--------|--------|----------|-----------|
| 折线图 | LTTB | 200:1 | 984ms | 235MB | 110MB ✅ |
| 柱状图 | 等宽聚合 | 200:1 | 1077ms | 257MB | 110MB ✅ |
| 箱线图 | 五数概括 | 4.5:1 | 2569ms | 445MB | 345MB |
| 散点图 | KDE 加权 | 200:1 | 1935ms | 325MB | 135MB ✅ |

**优化建议：**
- 折线/柱状/散点图：压缩效果优秀，内存回收及时
- 箱线图：因业务特性压缩比受限，建议控制组数（<1000组）
- 目前各类算法性能仍有提升空间，内存占用可进一步优化

## 四、遗留问题

1. 数据采样后可能会导致 echarts 功能部分异常，如 tooltip、legend、dataZoom、visualMap 等，需要确认采样后这些功能是否正常。
2. 采样后，实际渲染数据量大幅度减少，部分特征点可能丢失，如：极值点，异常点，用户标注点等，需要考虑如何识别并保留这些特征点。
3. 目前采样性能测试数据结构简单，且场景单一，需要进一步扩展测试场景，如：不同数据类型，不同数据量级，不同数据分布，复杂交互和视觉等等。
4. 考虑到定制化需求较多，部分 Echarts 无法实现效果可能考虑用 D3js 或其它库实现，后续需要考虑对接不同图表库时的性能表现差异。
5. 目前采样算法均为单线程实现，后续可以考虑多线程优化，提升采样性能。
6. 采样算法目前未覆盖 box chart，后续需要实现 box chart 的采样算法。
7. 目前降采样算法仅能满足宏观概览，即只关注整体趋势与异常区域定位，无法满足绝对精度的要求，因此后续需要考虑如何实现绝对精度的采样算法，初定技术方案：

    `1.宏观概览状态：自定义 Tooltip + 外部数据缓存，实现绝对精度数据展示。`
    
    `2.查看局部细节：动态采样切换，DataZoom 智能触发层级切换，原始数据直接渲染，消除采样不确定性。`
